## 双工具协作原则(写死,不要动)

1. **这个项目可能由 Claude Code 或 Codex 打开,两个工具必须共用同一套项目文件。**
2. **每一次实质改动都要立刻写回 `CONTEXT.md`,不能只留在对话里。** 不止"重要进展",**每次**改完一段就记一笔(动了哪些文件、为什么这么改、留给下次的尾巴)。关掉会话就丢的进度等于没做——很多 bug 就藏在被忽略的小改动里。
3. **优先更新 `CONTEXT.md` 和 `daily/[日期].md`。**
   - `CONTEXT.md` = 项目当前状态(正在做什么、卡在哪、下一步)
   - `daily/[日期].md` = 当天具体改动日记(可选,改动多时建)

## 启动流程

每次新会话开始,**按这个顺序读**:

1. 先读全局配置(`~/.claude/CLAUDE.md` 或 `~/.codex/AGENTS.md`,内容一样)。
2. 再读这份 `AGENTS.md`(项目规则)。
3. 再读 `CONTEXT.md`(上次进度)。
4. 才开始动手干活。

会话结束前**记得回写 `CONTEXT.md`**,把这次做了什么、下次接着做什么写清楚。

## 编码原则 (Karpathy 准则)

> 源自 Andrej Karpathy 总结的"避免 LLM 编码常见错误"的行为准则。
> **权衡**:这些规则偏谨慎、偏慢。琐碎任务可自行判断要不要严格走完整流程。

### 1. 动手前先思考 (Think Before Coding)

**不假设、不掩饰困惑、把权衡摆出来。**

实现前:
- 显式陈述假设。不确定就问。
- 多种合理解读时,列出来让用户挑,**不要默默选一个**。
- 看到更简单的方案,说出来。值得的话提出反对意见。
- 不清楚的地方,停下、点名困惑、问出来。

### 2. 简洁优先 (Simplicity First)

**用最少代码解决问题。不要做超出需求的事。**

- 不加用户没要的功能。
- 不为单次使用的代码做抽象。
- 不加未被要求的"灵活性"/"可配置性"。
- 不为不可能发生的场景做错误处理。
- 如果写了 200 行而其实 50 行就够,重写。

自问:"资深工程师会不会觉得这太复杂?" 会,就简化。

### 3. 外科手术式改动 (Surgical Changes)

**只动必须动的地方。只清理你自己制造的烂摊子。**

改动现有代码时:
- 不"顺手"改邻近代码、注释、格式。
- 不重构没坏的东西。
- 跟现有风格走,即使你觉得有更好的写法。
- 看到不相关的死代码,**提一下、别删**。

改动产生孤儿时:
- 删掉**你的改动**导致不再被使用的 import / 变量 / 函数。
- 不要顺手删原本就存在的死代码,除非被明确要求。

判定标准:每一行改动都能追溯到用户的需求。

### 4. 目标驱动执行 (Goal-Driven Execution)

**定义可验证的成功标准。循环到通过为止。**

把任务转成可验证目标:
- "加校验" → "为无效输入写测试,再让它过"
- "修 bug" → "先写能复现的测试,再让它过"
- "重构 X" → "改动前后测试都通过"

多步任务先列简短计划:

    1. [步骤] → 验证:[检查项]
    2. [步骤] → 验证:[检查项]
    3. [步骤] → 验证:[检查项]

强成功标准让我能独立循环;弱标准("让它工作")会逼着反复确认。

---

**这些原则起作用的迹象**:diff 里更少不必要的改动、更少因过度设计被迫重写、澄清问题在动手前出现而不是出错之后。
